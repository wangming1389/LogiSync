## 1. Design Constraints

### 1.1 Performance

- **Response time:** Most requests processed within 3–5s under normal conditions. Login to dashboard: ~2–3s. Credit validation: ~2–3s (synchronous). Product search: ~2–3s. Audit log query: ~5s.
- **Notification latency:** Order/approval notifications < 1 min. Driver dispatch push < 15s. Workspace result ~1 min. Dispute escalation alert < 1 min.
- **Throughput:** GPS ingestion at 1 ping/30s per active driver. Stable concurrent performance under typical operating conditions.
- **e-POD offline sync:** Data synchronized within 5–15s of network restoration. Offline capability up to ~24 hours. Zero data loss target.

### 1.2 Availability

- **Uptime:** >= 95%. MTTR < 2h. RPO < 24h. RTO < 24h.
- **Session management:** Workspace suspension invalidates sessions within 10–30s. Employee deactivation takes effect within 10–30s on next request. Token blacklisting enforced within 5–15s. Role/permission updates propagate within 10–30s without restart.
- **Fault detection:** Health checks every 15s. Failures trigger admin notification within 1–2 min. Critical issues (e.g., audit log write failures) detected and reported within 1–2 min.
- **Offline resilience:** Driver e-POD app supports offline operation using local encrypted storage. Data preserved and auto-synced on reconnect.

### 1.3 Security

- **Authentication:** bcrypt password hashing with salt. JWT-based stateless tokens with short expiration. Account lockout after multiple failed attempts. Global session invalidation after password changes.
- **Authorization:** Enforced at the API layer (RBAC middleware), never at the UI level. Resource-level access control on orders, disputes, e-POD. Break-glass access time-limited and logged. Strict domain separation: Supplier / Buyer / Carrier.
- **Data protection:** Registration records include immutable compliance fields (IP, timestamp, accepted terms version). Sensitive documents served exclusively via time-limited signed URLs (1h TTL). Direct file URLs never exposed in API responses.
- **Auditing:** Audit logs are append-only and cannot be modified after creation. All critical actions recorded for traceability.

### 1.4 Usability

- Input validation (e.g., tax ID) within 100–500ms on user interaction.
- Duplicate data (e.g., SKU) detected within 1–2s without page reload.
- Critical actions (e.g., workspace deletion) require confirmation; server-side validation within 3–5s.
- Location-based check-in validated server-side within 1–3s.
- Multi-step workflows (e.g., e-POD) enforced consistently.
- Role/permission updates reflected on subsequent user interaction (next page load within 2–3s).

### 1.5 Modifiability

- Business configurations (catalog data, KPI weights) can be updated dynamically, taking effect within 15–20s without restart.
- Price lists support versioning with effective timestamps.
- Pricing/rate updates create new versions without affecting existing records.

### 1.6 Data Integrity

- Critical operations (payment matching, credit updates) processed atomically within 4–5s.
- Data reconciliation (3-way matching) handled server-side within 3–5s.
- Background jobs (auto-confirmation) run every 5–15 min and are idempotent.
- Finalized records become immutable. Modification attempts return HTTP 409 Conflict within 4–5s. Corrections via append-only mechanisms only.

### 1.7 Scalability

- GPS route history stored for >= 5 years with archival strategy.
- File uploads validated at API: images <= 5MB, documents <= 10MB.
- Storage usage monitored per tenant; alerts at 80% capacity.
- Strict tenant isolation enforced at all layers. Verified by automated cross-tenant injection tests on every deployment.

## 3. Architectural Representation

### 3.1 Logical View

LogiSync follows a **Modular Monolith** architecture, encapsulating business logic into independent domain modules. This design optimizes performance via high-speed in-memory communication while maintaining strict boundaries for future migration to microservices.

**Layers:**

- **Clients Layer:** Multi-platform interfaces — web portals (Next.js SSR) for administrative roles and a React Native mobile app for field operations.
- **API Gateway & Security Layer:** Single entry point managing request routing, rate limiting, JWT validation, RBAC enforcement, and tenant isolation middleware — all enforced before any business logic.
- **Core Business Modules:**
  - IAM & Workspace Module
  - Sourcing & Order Module
  - Dispatch & Fleet Module
  - Delivery & e-POD Module
  - Finance & Credit Ledger Module
  - Cross-cutting services: Notification Service, Background Worker Pool, Audit Logger, GPS Ingestion Pipeline, Reputation Engine
- **Persistence Layer:** PostgreSQL + TimescaleDB, Redis, MinIO

### 3.2 Implementation View

**Backend (NestJS Modular Monolith):**

- Domain-driven modules: Finance, Dispatch, Sourcing, Delivery, IAM
- Internal communication via in-process typed function calls (eliminates network overhead, target < 500ms API response)
- Module boundaries enable future microservices extraction with minimal refactoring

**Frontend & Mobile:**

- Next.js (web): SSR for fast initial load, role-based routing, internal caching for catalog/search
- React Native (Driver App): Offline state machine, local encrypted storage (bounded <= 200MB), automatic delta-sync on reconnect

**Async Messaging:**

- RabbitMQ/Kafka Priority Queues: High-priority (driver push < 15s), Standard (order/workspace/dispute < 1 min), Low (reports/auto-confirmations)
- Background Worker Pool: e-POD image compression, PDF report generation, 48h auto-confirmation logic

**CI/CD:**

- GitHub Actions pipeline; mandatory cross-tenant injection tests block build on any isolation failure
- Static analysis, unit testing, and contract tests verifying Buyer-facing endpoints never return raw PII or internal storage paths

### 3.3 Deployment View

All services containerized via Docker Compose on a dedicated self-hosted server (VPS or bare-metal):

| Container                   | Role                                                                            | Key Constraints                                   |
| --------------------------- | ------------------------------------------------------------------------------- | ------------------------------------------------- |
| Backend API Container       | Synchronous transactional requests (Sourcing, Finance, IAM, Dispatch, Delivery) | Response target < 500ms, 20% spare capacity       |
| Background Worker Container | Async tasks — report generation, e-POD image compression, notification dispatch | Driver push SLA < 15s, isolated from main API     |
| GPS Ingestion Container     | High-throughput GPS ping reception into TimescaleDB                             | 1 ping/30s/driver sustained, isolated I/O         |
| MinIO Container             | Local S3-compatible private object storage                                      | Private buckets only, signed URL serving (1h TTL) |

**Persistence:**

- PostgreSQL + TimescaleDB: Primary relational engine with RLS for tenant isolation; TimescaleDB for GPS time-series with 5-year retention
- Redis: Session store, distributed cache, priority message broker queues. Workspace suspension enforced globally within 5–30s
- MinIO: Self-hosted object storage within Docker bridge network; all access via signed URLs

**Networking:** Private Docker bridge network — no public-facing database, cache, or storage ports exposed. Health-check polling every 15s; resource > 80% triggers admin alert within 1–2 min.

### 3.4 Data View

**PostgreSQL (Relational):**

- Tenant isolation via RLS using mandatory `workspace_id` on all tenant-scoped tables (except public product catalogs with `status = 'active'`)
- Append-only audit store — DB triggers populate immutable audit tables, forbidden UPDATE/DELETE, forensic retention >= 5 years
- Business record immutability via status-based `is_locked` flag with DB triggers
- Optimistic locking (version-based) for financial entities

**TimescaleDB (Time-Series):**

- High-frequency GPS ingestion (1 ping/30s per active driver)
- Automatic time-based partitioning; stable queries across 5-year history
- Append-only — UPDATE/DELETE rejected; legally defensible route history
- Tiered lifecycle: 90-day hot storage, then auto-archive >= 5 years

**MinIO (Object Storage):**

- All buckets configured as Private
- e-POD proofs, driver ID images (5MB limit), transport orders, invoices (10MB limit)
- All access via presigned URLs with 1h TTL; URL generation events logged atomically

**Redis (Cache):**

- Session blacklists and deactivation flags — suspension enforced globally within 5–30s
- Reputation score cache (pre-calculated) for sub-2s product search
- SKU index cache for real-time duplicate detection
- Priority message queues for notification dispatch

### 3.5 Process View

**Synchronous (Critical Logic):**

- 3-way matching, credit restoration, carrier selection + credit check processed synchronously on main execution thread
- Row-level locking in PostgreSQL for atomic transactions within 3–5s
- Idempotency keys on critical financial operations for safe retries

**Asynchronous (Three-Tier Priority Queuing):**

- High Priority: Transport Order push to drivers (guaranteed < 15s via FCM/APNs pre-authenticated connections; queue backlog > 30s triggers immediate alert)
- Standard Priority: Order approval/rejection, workspace results, dispute escalation alerts (< 1 min)
- Low Priority: 48h auto-confirmations, reputation recalculations, PDF report generation

**Resource Isolation:**

- GPS Ingestion Container isolated from main transactional DB
- Background Worker Container isolated from main API container
- MinIO I/O operations monitored to prevent bottleneck on PostgreSQL transaction logs

### 3.6 Scenario View

**Scenario 1: Transport Order Issuance & Instant Driver Notification**

1. Dispatcher issues order; RBAC validates Carrier domain permissions
2. API validates all Decree 10/2020 mandatory fields server-side before state transition
3. Atomic transaction commits order to PostgreSQL with `is_locked = true` + audit log entry
4. Order-Before-Signal: transport order persisted before notification enqueued
5. High-priority worker pool: driver receives push notification < 15s; Dispatcher UI confirmation within 3–5s

**Scenario 2: Offline e-POD Collection & Synchronization**

1. Driver offline: local state machine tracks 4-step progress; photos/signatures saved to local encrypted storage (bounded <= 200MB) via atomic writes; cryptographic tagging with Driver ID + device timestamp
2. On reconnect: delta-sync with sequence numbering; server validates sequence integrity and retroactively validates geofence check-in coordinates
3. Validated files streamed to MinIO Container; e-POD records written to PostgreSQL with `is_locked = true`; audit log committed atomically
4. Signed URL (1h TTL) generated for dispatcher review; shipment status set to `Completed`; full sync within 5–15s

**Scenario 3: Automated Credit Limit Restoration**

1. Accountant matches payment receipt against invoice
2. Finance module initiates synchronous in-process reconciliation with row-level locking on Credit Ledger
3. Atomic transaction: append-only payment record + credit balance restoration + audit log (old balance, new balance, invoice reference, actor, IP, timestamp)
4. Idempotency key prevents double-restoration on retry
5. Credit limit restored and visible to Buyer within < 3s
